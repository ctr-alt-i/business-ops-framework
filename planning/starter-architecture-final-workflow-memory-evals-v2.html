Final Starter Architecture + Pre-Build Package

Final revision based on the earlier architecture work, the LLM wiki pattern, the Accountilytics structure review, the clarified product direction, and the original business-ops-framework-features.html requirements source. This should not be framed as just a wiki framework. It should be framed as a workflow framework powered by an LLM-maintained memory layer, with the features document distilled into a smaller v1 build scope.

Final product framing: build an LLM-native business operations workflow framework that uses managed wiki workspaces as persistent memory for ingest, synthesis, reporting, and operational workflows.

1. Final architecture thesis

The wiki is not the end product. The wiki is the memory substrate that allows workflows to become more useful over time. The workflows are the user-facing product layer.

Source layer

Raw, immutable business evidence: transcripts, emails, docs, requirements, decks, invoices, legal records, and other source material.

Memory layer

LLM-maintained wiki pages, indexes, logs, summaries, entity pages, project pages, and ops pages that compile source material into durable context.

Workflow layer

Reusable workflow definitions that pull from memory and sources to create operational outputs, suggested actions, updates, and automations.

Interface layer

Pi first, then later schedulers, APIs, and possibly a web app. The interface should call the framework, not contain it.

raw sources -> wiki memory -> workflows -> outputs / actions
Final design rule: the wiki exists to support workflows, and the workflows exist to turn memory into action.

4. First-class modules to build

Module Purpose Priority
workspace Initializes and validates workspaces, manages schema files, and enforces path conventions. Phase 1
ingest Reads raw sources, classifies them, creates source summaries, and orchestrates wiki updates. Phase 1
wiki Creates and updates pages, maintains links, index, log, and lint behavior. Phase 1
operations Extracts decisions, approvals, actions, future work, and related operational facts. Phase 2
workflows Defines workflow specs, triggers, context builders, steps, outputs, and run logs. Phase 2
outputs Generates weekly reports, meeting recaps, status packets, and other artifacts from memory and sources. Phase 2
pi adapter Exposes framework use-cases into Pi commands, tools, and skills. Phase 3
external adapters Imports email, transcripts, docs, calendar, and messaging context into the raw source layer. Phase 4

5. Updated development order

  1. Lock the product framing and v1 scope: use the distilled feature scope from business-ops-framework-features.html to confirm workspace scope, first source type, and first workflow set.
  2. Create the repo skeleton: initialize the repository with src/, workspaces/, evals/, resources/, and apps/pi-console/.
  3. Implement the workspace module: init, validate, and load workspace structure plus AGENTS.md.
  4. Implement one-source-at-a-time ingest: start with meeting transcripts or a similarly high-value source.
  5. Implement wiki maintenance: source summaries, index.md, log.md, overview.md, and a few durable page categories.
  6. Implement operations extraction: decisions, action items, approvals, and future work.
  7. Implement the workflow runtime: workflow definitions, runner, run logging, context assembly, and manual triggers.
  8. Implement the first outputs: meeting recap, weekly report, and a simple daily briefing or status summary.
  9. Wire Pi into the framework: keep Pi thin and let it trigger the use-cases above.
  10. Add linting and health checks: stale claims, contradictions, missing pages, and missing links.
  11. Add the first integrations only after the file-based workflow works: transcript import, email import, and calendar context are the best first candidates.
  12. Add scheduled runs and retrieval upgrades last: only after the manual loop is already useful.
Recommended build sequence in one line: scope -> repo -> workspace -> ingest -> wiki -> extraction -> workflows -> outputs -> Pi -> adapters -> automation.

6. The artifacts you should have before the next build session

Before starting a fresh implementation session and initializing the git repo, I would want a small but complete planning packet. These artifacts do not need to be perfect. They just need to be clear enough to scaffold against and evaluate against.

Artifact What it should contain Why it matters Priority
1. Product framing note One short document stating what the framework is, what it is not, and what v1 is trying to prove. Keeps the project from drifting between “wiki tool,” “agent shell,” and “workflow platform.” Must have
2. Features source reference The original business-ops-framework-features.html kept in the planning packet as the source requirements document. This preserves the full product ambition and prevents the architecture from narrowing the product too much. Must have
3. V1 feature scope distillation A compressed v1-feature-scope.md derived from the features doc with sections for must-have, next, later, non-goals, and success criteria. Turns a large feature inventory into a buildable scope for scaffolding and eval design. Must have
4. Workspace scope decision Whether a workspace equals a client, an engagement, or a project. This affects paths, naming, and eval design. Must have
5. V1 workflow definitions Short specs for the first 2–3 workflows, ideally post-call processing, weekly reporting, and morning briefing. Workflows are the product layer; they should shape the build early. Must have
6. Workspace schema draft A draft AGENTS.md or schema note that defines folder rules, page conventions, ingest behavior, and naming. This turns the LLM into a disciplined maintainer. Must have
7. Wiki page template set Examples for source summary pages, entity pages, project pages, overview pages, and ops pages. Prevents structure thrash during implementation. Must have
8. Operational extraction schema Definitions for decision, approval, action item, future work, stakeholder, and risk objects. Needed for extraction and eval consistency. Must have
9. Sample raw corpus A small but realistic set of source files: transcripts, notes, technical docs, maybe an email thread or requirement doc. You need real inputs to build the scaffolding around. Must have
10. Expected output examples At least one gold example each for a meeting recap and a weekly report. These become golden references for evals. Must have
11. Naming and frontmatter conventions Simple rules for filenames, dates, IDs, tags, and markdown frontmatter. Keeps the workspace legible and Obsidian-friendly. Should have
12. Evals plan The first set of eval cases, pass/fail criteria, and where fixtures and goldens will live. Lets development proceed with regression protection from the start. Must have
How to treat the features doc: keep business-ops-framework-features.html as the full requirements source, but do not use it as the raw implementation backlog. Distill it into v1-feature-scope.md so the scaffold and evals stay focused.

8. Evals strategy from day one

Since you want to begin development with evals, I would make evals part of the architecture instead of something added later.

8.1 What to evaluate first

  • Ingest evals: does a source create the expected summary page, log entry, and index update?
  • Extraction evals: are decisions, actions, approvals, and future work correctly captured from a known input?
  • Wiki update evals: are the right pages touched, linked, or created?
  • Output evals: does the workflow generate an acceptable meeting recap or weekly report?
  • Lint evals: can the system detect obvious contradictions, missing links, or stale pages in a controlled fixture?

8.2 Recommended eval folder shape

evals/
├─ fixtures/
│  ├─ raw/
│  └─ workspaces/
├─ goldens/
│  ├─ wiki/
│  └─ outputs/
├─ suites/
│  ├─ ingest/
│  ├─ extraction/
│  ├─ wiki/
│  ├─ workflows/
│  └─ lint/
└─ rubrics/
   ├─ meeting-recap.md
   └─ weekly-report.md

8.3 Goldens vs rubrics

Use goldens when structure matters

For index updates, decision extraction, action items, page creation, and deterministic fields.

Use rubrics when quality matters

For recaps, reports, summaries, and other outputs where exact text match is too strict.

Eval rule: every new workflow should come with at least one fixture, one expected wiki effect, and one expected output artifact.
