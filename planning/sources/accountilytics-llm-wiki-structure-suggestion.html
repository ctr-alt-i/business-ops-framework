LLM Wiki / Project Folder Structure Suggestion

Context reviewed: temp/llm-wiki.md and the Google Drive folder Accountilytics X Ctrl.Alt.I.

Goal: suggest a very simple, framework-friendly folder model that follows the LLM wiki pattern without overdesigning v1.

1) Gist of llm-wiki.md

Persistent wiki, not just retrieval

The key idea is that the LLM should not rediscover knowledge from raw files every time. Instead, it should maintain a living wiki that accumulates summaries, entities, concepts, links, contradictions, and synthesis over time.

Three-layer model

Raw sources are immutable. The wiki is LLM-authored and continuously updated. The schema tells the LLM how to behave, where things go, and how ingest/query/lint should work.

Start simple

The document explicitly says the exact structure should fit the domain. It recommends simplicity first: an index, a log, and a disciplined workflow before adding fancy tooling.

Core insight

The current structure is organized mostly by document type or business function. The LLM wiki model works better when organized by content role in the workflow:

  • What comes in → raw sources
  • What the LLM maintains → wiki
  • What humans send/share/use → outputs
  • What must be kept for business/admin reasons → admin
  • What is no longer active → archive

3) Recommended v1 structure: simple on purpose

My recommendation: use only 5 top-level directories plus one schema file at the root.

  README.md

  admin/
    legal/
    finance/

  raw/
    meetings/
    technical/
    clients/

  wiki/
    index.md
    log.md
    overview.md
    topics/
    entities/
    projects/

  outputs/
    reports/
    decks/

  archive/

admin/

For contracts, invoices, privacy docs, payment records, and any non-wiki business paperwork.

raw/

For immutable input material. This is the source of truth the LLM reads from but does not rewrite.

wiki/

The missing layer today. This is where the LLM builds durable knowledge: summaries, entities, decisions, systems, workflows, and synthesis pages.

outputs/

For reports, decks, client-facing materials, and shareable artifacts generated from the wiki or project work.

archive/

For retired experiments, duplicates, old exports, and superseded materials.

AGENTS.md

The schema/instruction file that tells the LLM how to ingest raw material, update the wiki, answer questions, and keep the structure consistent.

4) Why this is the right level of simplicity

  • It directly mirrors the LLM wiki concept: raw → wiki → outputs.
  • It keeps admin separate so the wiki does not become a junk drawer.
  • It avoids too many top-level file-type folders like Meeting Transcripts, Technical Docs, and Weekly Reports.
  • It is reusable as a framework across clients and projects.
  • It leaves room to grow without forcing complexity on day one.

Important constraint for v1

Do not add more top-level directories until there is repeated pain.

The framework should prefer subfolders inside existing roles before creating a new root category.

6) Suggested approach for understanding the core directories in any future project

  1. Inventory the current folders and list representative files.
  2. Classify by workflow role, not by file type: incoming source, knowledge layer, output, admin, archive.
  3. Ask what the LLM should own. If the LLM should continuously rewrite it, it belongs in wiki/.
  4. Ask what must remain immutable. If it is a source or evidence artifact, it belongs in raw/.
  5. Collapse similar directories. For example, decks + reports are both outputs; invoices + legal docs are both admin.
  6. Create new top-level folders only when behavior differs materially. If two categories have the same lifecycle, they probably do not need separate roots.

A practical test

If a folder answers one of these questions, it is probably core:

  • Where do new materials land? → raw/
  • Where does synthesized knowledge live? → wiki/
  • Where do client-ready artifacts go? → outputs/
  • Where do contracts and financial records live? → admin/
  • Where do old things go? → archive/
